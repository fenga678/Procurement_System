# 代码评估与重构路线图

## 1. 目的与原则

本文档用于固化当前项目的代码评估结论和后续重构执行路线，作为后续优化工作的统一依据。

本次优化明确遵循以下原则：

1. 在功能不变的前提下进行重构。
2. 不改变现有业务规则、界面交互结果和数据含义。
3. 优先解决影响稳定性、流畅度和维护成本的问题。
4. 所有重构应尽量小步推进，并具备回归验证手段。
5. 优化顺序按收益、风险和实施难度综合排序。

---

## 2. 当前代码总体评估

### 2.1 总体结论

当前项目已经具备较完整的业务原型能力，核心业务流程、桌面界面、数据库访问、分类与商品维护、采购任务维护、采购明细落库等功能链路基本齐全。

但从工程实现角度看，当前项目更接近“可运行的业务原型”而不是“稳健的生产级桌面应用”。主要问题集中在以下几个方面：

1. 架构分层意图存在，但落地不彻底。
2. 核心算法与真实数据源存在脱节风险。
3. WPF 层承担了过多数据访问和事务职责。
4. 授权模块存在明显安全和副作用问题。
5. 存在源码乱码和文本编码污染。
6. 自动化测试不足，缺乏重构保护网。
7. 数据库初始化和升级机制不够规范。

### 2.2 综合评分

基于当前代码状态，给出以下主观评分：

| 维度 | 评分 | 说明 |
| --- | --- | --- |
| 业务原型能力 | 7/10 | 业务链路比较完整，具备实际使用基础 |
| 架构落地质量 | 4/10 | 分层结构存在，但实现边界混乱 |
| 可维护性 | 4/10 | 耦合偏高，乱码和静态依赖明显 |
| 稳定性 | 4/10 | 有事务控制，但异常边界和升级机制较弱 |
| 生产可用性 | 3/10 | 仍需较多工程化加固才能稳定长期运行 |

## 3. 主要问题评估结果

以下问题按影响程度排序。

### 3.1 核心算法未真正接入真实业务数据

#### 问题描述

核心排程算法内部存在硬编码分类和商品数据的实现方式，导致系统中维护的分类、商品、频率、价格等数据不一定真正参与采购计划生成。

#### 影响

1. 界面维护的数据与算法输出可能不一致。
2. 用户以为修改了配置，实际上排程结果未真正受影响。
3. 后续优化算法时容易形成“双份数据逻辑”。
4. 属于直接影响业务正确性的高优先级问题。

#### 风险判断

这是当前最严重的问题之一。若不优先修复，任何围绕算法的体验优化都可能建立在错误数据基础上。

### 3.2 分层结构名义存在，但实现层次穿透严重

#### 问题描述

项目目录中存在 `Core`、`Application`、`Infrastructure`、`Wpf` 等层次，但实际实现上：

1. `Wpf` 层直接管理数据库连接、SQL、事务和 schema 检查。
2. `Infrastructure` 项目基本未承担实际基础设施职责。
3. `Application` 层主要停留在接口与 DTO 定义，缺少真正的用例编排。

#### 影响

1. UI 与数据访问强耦合。
2. 一旦更换数据库实现或调整数据结构，页面代码会被迫大量修改。
3. 页面代码难以测试，复用性差。
4. 长期维护时容易出现逻辑散落、多处重复、缺少统一约束。

### 3.3 WPF 页面层承担过多职责

#### 问题描述

当前多个服务类位于 `Wpf\Services` 下，直接执行查询、保存、更新、删除、事务控制、schema 判断、批量写入等操作。

#### 影响

1. UI 层过重，页面代码容易进一步膨胀。
2. 页面响应速度容易受数据库 I/O 拖累。
3. 不利于引入统一日志、统一重试、统一异常处理。

### 3.4 授权模块安全性不足，且存在副作用

#### 问题描述

当前授权模块存在以下问题：

1. 客户端硬编码 `SecretKey`。
2. 注册码本质上是可逆的 `Base64(JSON)`。
3. 签名逻辑只是哈希截断，不属于可靠签名方案。
4. AES 使用固定零 IV。
5. 读取授权状态时存在触发保存行为的副作用。

#### 影响

1. 授权保护容易被绕过。
2. 状态验证逻辑可读性差，维护成本高。
3. 本地状态文件可能被反复覆盖，增加不可预期行为。

#### 风险判断

如果软件对外分发，该模块应视为中高风险，需要在功能保持不变的前提下尽快重构实现方式。

### 3.5 源码乱码与编码污染

#### 问题描述

多个 C# 文件中的中文注释、日志文本、异常消息、状态描述已经出现乱码。

#### 影响

1. 直接影响维护效率。
2. 异常排查困难。
3. 可能影响最终用户提示文本。
4. 不利于团队协作和后续交接。

### 3.6 缺乏自动化测试

#### 问题描述

当前测试目录基本未形成有效覆盖，至少没有足够的核心业务测试作为重构护栏。

#### 影响

1. 任何重构都需要依赖人工点测。
2. 容易在优化性能或拆分结构时引入隐性回归。
3. 采购算法、预算规则、频率规则等关键逻辑缺乏可重复验证手段。

### 3.7 数据库初始化和升级机制不规范

#### 问题描述

SQLite 当前主要通过“检查某张表是否存在”来判断数据库是否完成初始化，缺少正式版本迁移机制。

#### 影响

1. 不能可靠处理增量升级。
2. 旧版本数据库可能出现列缺失、索引缺失、结构不一致问题。
3. 用户环境升级失败时难以定位和修复。

## 4. 优化目标

在功能不变前提下，本轮重构要达成以下目标：

### 4.1 性能目标

1. 页面加载、切换、查询、保存时更流畅。
2. 长耗时操作不阻塞 UI 主线程。
3. 降低数据库访问次数和不必要的对象创建。

### 4.2 稳健性目标

1. 常见异常不导致程序闪退。
2. 数据写入过程更可靠，避免脏数据和半成功状态。
3. 数据库版本升级可控，失败可定位。

### 4.3 维护性目标

1. 逐步让 WPF 层专注界面职责。
2. 让应用层和基础设施层真正承担对应角色。
3. 为后续继续演进留下明确边界。

---

## 5. 重构总策略

整体采用“四步推进”策略：

1. 先修稳定性，再做性能优化。
2. 先补保护网，再做结构拆分。
3. 先做低风险高收益项，再动深层架构。
4. 每个阶段完成后必须进行回归验证。

即：

1. P0 修稳定性与基础质量。
2. P1 做流畅度与性能优化。
3. P2 做架构解耦与职责收敛。
4. P3 做数据库治理与配置治理。
5. P4 做授权与安全加固。

---

## 6. 按优先级排序的重构路线图

## P0 核心稳定性与基础质量

这是最高优先级阶段。完成后，系统才具备继续做结构重构和性能优化的安全基础。

### P0-1 修复源码编码和乱码问题

#### 目标

统一文本编码，消除源码和用户提示中的乱码问题。

#### 实施步骤

1. 盘点所有 `.cs`、`.xaml`、`.json`、`.sql`、`.md` 文件的编码格式。
2. 将源码统一转换为 `UTF-8`。
3. 修复所有乱码的异常提示、日志文本、注释和授权状态文本。
4. 优先处理以下目录：
   - `src\CanteenProcurement.Core`
   - `src\CanteenProcurement.Wpf\Services`
   - `src\CanteenProcurement.Wpf\Providers`
5. 建立编码规范，后续提交统一使用 `UTF-8`。

#### 验收标准

1. 代码文件不再出现乱码文本。
2. 程序中用户可见提示文本可正常显示中文。
3. 日志与异常信息可直接阅读。

### P0-2 建立统一异常边界与错误反馈

#### 目标

避免异常直接冒泡导致 UI 卡死或闪退。

#### 实施步骤

1. 盘点所有页面入口操作：
   - 页面初始化
   - 查询刷新
   - 新增/编辑保存
   - 删除确认
   - 生成采购计划
   - 导入导出
2. 为所有入口添加统一异常捕获。
3. 区分异常类型：
   - 用户输入校验错误
   - 数据库连接错误
   - 数据完整性错误
   - 配置错误
   - 授权错误
   - 未知系统错误
4. 为统一反馈服务补充标准化提示文案。
5. 对关键异常增加日志记录，但避免泄露敏感信息。

#### 验收标准

1. 常见操作出错时程序不闪退。
2. 用户能收到可理解的错误提示。
3. 日志可用于定位失败原因。

### P0-3 加强事务一致性和数据有效性保护

#### 目标

保证写入操作在异常情况下不留下脏数据或半成功状态。

#### 实施步骤

1. 逐个审查所有数据库写入操作：
   - 新增分类
   - 更新分类
   - 删除分类
   - 新增商品
   - 更新商品
   - 删除商品
   - 创建任务
   - 更新任务状态
   - 替换任务明细
   - 保存固定金额预算
2. 对所有删除后重建型操作保留事务包裹。
3. 提升输入校验前置化，尽量在入库前拦截异常数据。
4. 对批量插入增加结果校验和失败回滚。
5. 对数据库返回值为 0 的关键操作明确失败语义。

#### 验收标准

1. 所有关键写入流程都有清晰事务边界。
2. 异常退出后数据库状态保持一致。
3. 无效明细、超界值、缺字段数据不能写入数据库。

### P0-4 建立最小测试保护网

#### 目标

在不大幅增加开发负担的前提下，为后续重构建立最基础的回归保护。

#### 实施步骤

1. 新建测试项目并接入解决方案。
2. 先补业务价值最高的测试：
   - 预算分配测试
   - 频率调度测试
   - 最小间隔规则测试
   - 浮动范围测试
   - 固定金额分类测试
3. 增加基础校验测试：
   - 分类校验
   - 商品校验
   - 任务校验
4. 增加 1 组 SQLite 集成测试：
   - 建库
   - 新增
   - 查询
   - 删除
   - 明细替换

#### 验收标准

1. 核心算法和校验逻辑有基础测试覆盖。
2. 后续重构可通过测试快速发现明显回归。

## P1 性能与流畅度优化

该阶段目标是让用户明显感受到程序“更顺、更快、不卡”。

### P1-1 消除 UI 线程阻塞

#### 目标

让数据库查询、计划生成、保存操作不再阻塞主线程。

#### 实施步骤

1. 梳理所有长耗时操作是否在 UI 线程直接执行。
2. 所有查询、保存、生成、导入等操作改为异步执行。
3. 页面首次加载采用：
   - 立即显示页面框架
   - 同步显示 Loading
   - 异步加载数据
   - 加载完成后刷新绑定
4. 对长时间任务添加取消控制 `CancellationToken`。
5. 对按钮点击添加防重入保护，防止重复提交。

#### 验收标准

1. 查询时窗口不假死。
2. 保存和生成期间界面可响应。
3. 重复点击不会导致多次提交。

### P1-2 减少数据库访问次数

#### 目标

减少无效查询和重复连接带来的性能损耗。

#### 实施步骤

1. 盘点页面初始化阶段的查询链路。
2. 对 schema 能力检查结果做缓存。
3. 避免同一页面操作中多次重复打开连接。
4. 能合并的查询尽量合并，能并行的读取尽量并行。
5. 列表页只取列表所需字段，详情页再加载全部字段。

#### 验收标准

1. 页面首屏查询次数减少。
2. 连接打开次数明显下降。
3. 热点页面加载时间缩短。

### P1-3 优化批量写入和查询性能

#### 目标

让任务明细保存、任务详情查询、列表查询在数据量增大后仍保持平稳。

#### 实施步骤

1. 将批量插入的批大小提取为可配置常量。
2. 复核现有索引是否覆盖以下高频条件：
   - `created_at`
   - `purchase_date`
   - `category_code`
   - `product_id`
   - `status`
3. 为常用复合查询增加复合索引。
4. 对任务详情、采购明细、商品列表查询做执行路径评估。
5. 保持 SQL 结果字段精简，减少无意义字段读取。

#### 验收标准

1. 批量保存耗时下降。
2. 查询大列表时响应更快。
3. 数据量增长后性能退化变缓。

### P1-4 减少不必要对象创建和重复计算

#### 目标

降低 GC 压力和业务执行过程中的额外开销。

#### 实施步骤

1. 将规则注册从“每次执行时检查”优化为稳定生命周期初始化。
2. 减少同一流程内的重复分类、商品数据构建。
3. 避免页面频繁 new 大对象和服务对象。
4. 对频繁使用但不变化的数据做短生命周期缓存。

#### 验收标准

1. 采购计划生成耗时下降。
2. 页面频繁切换时卡顿减少。

## P2 架构解耦与职责收敛

该阶段是中期重点，目标是让当前代码从“能跑”过渡到“可长期维护”。

### P2-1 将数据访问逐步迁出 WPF 层

#### 目标

让 `Wpf` 层不再直接承担核心数据库访问职责。

#### 实施步骤

1. 先从 `CategoryDataService`、`ProductDataService`、`TaskDataService` 下手。
2. 抽出接口并放入 `Application` 层或独立契约层。
3. 在 `Infrastructure` 中实现具体仓储或查询服务。
4. 保持现有方法签名尽量不变，先迁移职责，再考虑进一步抽象。
5. 所有 SQL 逐步从 WPF 层迁走。

#### 验收标准

1. WPF 层不再直接包含大量核心 SQL。
2. 数据访问职责主要位于 `Infrastructure`。

### P2-2 让 Application 层承担用例编排职责

#### 目标

让应用层真正成为业务流程编排入口，而非仅停留在接口和 DTO。

#### 实施步骤

1. 将以下行为整理为应用服务：
   - 创建采购任务
   - 生成采购计划
   - 保存采购明细
   - 保存分类设置
   - 保存商品设置
2. 将业务校验从 UI 层下沉到应用服务层。
3. 将页面的“取值、校验、调用、提示”流程分清边界。
4. 统一输出 DTO，避免页面直接依赖底层实体和数据库记录类。

#### 验收标准

1. 主要业务流程都有清晰的应用服务入口。
2. 页面 code-behind 职责明显减轻。

### P2-3 收敛全局静态依赖

#### 目标

降低隐藏耦合，提高可测试性和可替换性。

#### 实施步骤

1. 梳理当前静态依赖：
   - `DatabaseConfig`
   - `LicensingService.Instance`
   - `MainWindow.Instance`
2. 从 `App` 启动入口引入简单依赖注入。
3. 页面通过构造注入使用服务。
4. 将全局可变状态限制在极小范围内。

#### 验收标准

1. 大部分页面和服务不再直接访问静态单例。
2. 服务可独立测试和替换。

## P3 数据库与配置治理

该阶段主要解决“长期运行、持续升级、跨环境部署”的稳健性问题。

### P3-1 引入正式数据库迁移机制

#### 目标

让数据库升级可追踪、可重复、可验证。

#### 实施步骤

1. 增加数据库版本记录表，例如 `schema_version`。
2. 所有结构变更都以增量脚本方式落地。
3. 启动时检查数据库版本，并按顺序执行迁移。
4. 对迁移过程增加日志和失败提示。
5. 将 `database` 目录下脚本按版本编号管理。

#### 验收标准

1. 旧数据库可按版本逐步升级到最新。
2. 升级失败时可以明确定位具体脚本。

### P3-2 统一配置管理

#### 目标

减少配置逻辑散落，提高运行参数的可控性。

#### 实施步骤

1. 统一管理数据库配置、批量写入大小、超时时间、日志级别等参数。
2. 明确哪些配置支持热更新，哪些需要重连或重启。
3. 避免多个服务分别维护各自的配置逻辑。
4. 为配置读取失败提供清晰回退策略和错误提示。

#### 验收标准

1. 配置入口统一。
2. 配置行为清晰、可预测。

---

## P4 授权与安全加固

该阶段不改变现有授权功能表现，但优化底层实现的稳健性和安全性。

### P4-1 重构授权验证逻辑

#### 目标

消除授权模块中的明显安全短板和副作用。

#### 实施步骤

1. 分离“注册码验证”和“注册码保存”逻辑。
2. 让授权状态读取成为纯读操作，不带写文件副作用。
3. 将当前伪签名机制升级为真正的公钥验签方案。
4. 保持用户侧使用方式不变。

#### 验收标准

1. 状态读取不再写入本地文件。
2. 授权验证逻辑清晰可审计。

### P4-2 保护敏感信息

#### 目标

减少敏感数据在源码、日志和配置中的裸露。

#### 实施步骤

1. 避免在日志中输出完整连接字符串。
2. 避免明文输出注册码、机器码、数据库密码。
3. 敏感配置使用更安全的本地存储方式。

#### 验收标准

1. 敏感信息不会直接暴露在日志和提示中。

## 8. 每阶段交付标准

### P0 交付标准

1. 所有源码无乱码。
2. 常见失败路径不闪退。
3. 核心算法和基础校验已有最小测试覆盖。

### P1 交付标准

1. 页面加载和切换时无明显卡顿。
2. 查询、保存、生成不阻塞 UI。
3. 热点页面和热点查询响应时间明显下降。

### P2 交付标准

1. WPF 层不再直接承担核心数据访问职责。
2. Application 层具有真实业务编排能力。
3. 静态全局依赖显著减少。

### P3 交付标准

1. 数据库支持版本化迁移。
2. 配置读取与变更行为统一、可控。

### P4 交付标准

1. 授权验证无明显安全短板。
2. 授权和敏感配置处理逻辑更稳健。

## 9. 实施过程中的硬性要求

为保证后续严格执行，后续每次重构必须遵守以下要求：

1. 不允许在未经验证的情况下改动业务规则。
2. 每完成一个子阶段，必须进行回归验证。
3. 重构前先补测试或先建立最小人工验证清单。
4. 所有新增代码保持统一编码、命名和异常处理风格。
5. 所有数据库结构变更必须同步记录到迁移脚本或迁移说明中。
6. 所有性能优化必须以“功能输出不变”为前提。
7. 避免一次性大改，优先分阶段、小批量合并。

## 11. 最终结论

当前项目已经具备较强的业务基础，但还没有达到“长期稳定维护、性能平稳、结构清晰”的状态。

在功能不变的前提下，最合理的路径不是立即大规模推倒重写，而是严格按照本文档的优先级顺序，分阶段完成：

1. 基础质量修复
2. 流畅度优化
3. 架构解耦
4. 数据库治理
5. 授权加固

只有这样，才能在不破坏现有业务功能的情况下，让程序逐步变得更流畅、更稳健、更易维护。
